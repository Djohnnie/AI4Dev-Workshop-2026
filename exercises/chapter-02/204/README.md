Finding bugs in someone else's code — or in code generated by AI — is one of the most valuable skills a developer can have. This exercise contains a working implementation of **Dijkstra's shortest-path algorithm** with one or more subtle bugs deliberately introduced. The unit tests expose the failures; your job is to use Copilot's `/fix` command and targeted prompting to locate and repair the defects.

The exercise also teaches a key prompting principle: **the quality of Copilot's fix depends directly on the quality of context you provide**. A vague prompt yields a vague fix; a precise prompt describing the expected behaviour leads to a surgical correction.

## 📚 Background: Dijkstra's Algorithm

**Dijkstra's algorithm** finds the shortest path from a single source node to every other node in a weighted, directed graph where all edge weights are non-negative.

### Core Idea

Maintain a set of "visited" nodes and a distance table. Repeatedly pick the unvisited node with the smallest known distance, then *relax* (potentially improve) the distances to its unvisited neighbours.

### Step-by-Step

Given a graph with nodes and weighted edges:

1. Set `distance[source] = 0`; set `distance[all others] = ∞`.
2. Add the source to a **min-priority queue**.
3. While the priority queue is not empty:
   - Dequeue the node `u` with the smallest tentative distance.
   - If `u` has already been finalized, skip it.
   - Mark `u` as finalized.
   - For each neighbour `v` of `u` with edge weight `w`:
     - If `distance[u] + w < distance[v]`, set `distance[v] = distance[u] + w` and enqueue `v`.
4. Return the `distance` table.

### Complexity

| | Value |
|-|-------|
| Time | O((V + E) log V) with a binary heap priority queue |
| Space | O(V + E) |

where V = number of vertices, E = number of edges.

### Why Non-Negative Weights Only?

Dijkstra relies on the guarantee that once a node is finalized, its distance cannot improve. A negative edge could create a path that improves a finalized distance, violating this invariant. For graphs with negative weights, use the **Bellman-Ford algorithm** instead.

### Common Bug Patterns

When implementing Dijkstra, developers frequently introduce these mistakes:

| Bug | Symptom |
|-----|---------|
| Wrong comparison operator in relaxation | Longer paths accepted, shorter ones ignored |
| Priority queue ordering inverted (max-heap instead of min-heap) | Always explores the farthest node first → wrong results |
| Missing finalization guard | Nodes relaxed multiple times → infinite loops or wrong distances |
| Off-by-one in adjacency list indexing | Some edges never explored |
| Initial distance set to `0` instead of `∞` for non-source nodes | All distances appear as 0 |

## ✅ Your Task

### Phase 1 — Discover

1. Run the tests without looking at the code first:

```bash
cd 204/ShortestPath.Tests
dotnet test
```

Note which tests fail and what the expected vs. actual values are.

### Phase 2 — Fix with Copilot

2. Open `ShortestPath/Dijkstra.cs`.
3. Select the failing method and open **Copilot Chat**.
4. Try each of these prompting strategies and observe the quality of the suggested fix:

   **Vague prompt (baseline):**
   ```
   /fix
   ```

   **Better — include the failure:**
   ```
   /fix The test for a diamond graph expects the shortest path A→D to be 3 
   (via A→C→D) but the method returns 4 (via A→B→D). 
   ```

   **Best — include algorithm context:**
   ```
   /fix This is Dijkstra's algorithm. The relaxation step is producing 
   incorrect distances for graphs where the greedy first hop is longer 
   than an indirect route. The test 'DiamondGraph_ReturnsCorrectDistances' 
   fails with expected 3 but got 4 for node D.
   ```

5. Compare the quality and precision of each fix Copilot suggests.

### Phase 3 — Verify

6. Apply the fix and re-run the tests until all pass.

## 💡 Prompting Lesson

This exercise demonstrates the **Context → Precision → Quality** chain:

1. The more context you give Copilot (failing test name, expected vs. actual, algorithm name), the more surgical its suggestion.
2. A `/fix` with no context produces a generic, often incorrect rewrite.
3. A `/fix` with the exact failure and the algorithmic invariant that is being violated produces a one-line targeted fix.

This principle applies to every Copilot interaction, not just bug-fixing.
